## ADR content prescriptions

What each ADR must decide. These are the review bar for P2–P7; the per-task
goals in `docs/roadmap/p5-tasks/README.md` (W0 section) are the contract.

**ADR-0055 — cloud credential + normalization model (shared).** Vault
credential kinds + JSON shapes for AWS (access key; optional STS assume-role)
and Azure (service principal), stored via D11 only; rotation integration per
ADR-0040; documented least-privilege read-only IAM policy / Azure role for
operators; `CLOUD_NETWORK_INVENTORY` capability signature (name PROPOSED);
normalized cloud models (virtual network, subnet, route table,
interconnect/peering, NIC) with the cross-provider equivalence table
(VPC↔VNet, SG↔NSG, TGW/peering↔VNet-peering, ENI↔NIC); SG/NSG→
`FIREWALL_POLICY` normalization contract incl. any-any and port-range
semantics (reuse the P2 ACL-parser disambiguation lessons); explicit
statement: both plugins read-only in P5, future cloud writes CR-gated.

**ADR-0056 — AWS plugin.** boto3 per D7; session/credential handling citing
0055 (no re-decision); region enumeration strategy; pagination + throttling
retry/backoff; recorded-fixture strategy (botocore-style stubbed responses,
raw payloads verbatim); `DISCOVERY_API` mapping table payload→normalized
model for VPCs/subnets/route tables/TGW+peering/SGs/ENIs; Route53 as
`DDI_DNS` through the existing DDI framework (zones/records → DDI models),
completing the BlueCat/Infoblox/Route53 triad.

**ADR-0057 — Azure plugin.** azure SDK per D7 (identity + mgmt-network);
service-principal auth citing 0055; subscription/resource-group enumeration;
fixture strategy; `DISCOVERY_API` mapping for VNets/subnets/route
tables/peerings/NSGs/NICs onto the SAME normalized models — any Azure-only
divergence needs a recorded reason.

**ADR-0058 — hybrid topology stitching.** Cloud node/edge kinds + PG schema
sketch (expand-only when implemented); Neo4j projection mapping; stitching
sources per provider (VPN gateway / Direct Connect / ExpressRoute); join
evidence rules (gateway endpoints + CIDR/route-table match); per-source
provenance format (ADR-0052-consistent); idempotency + retraction contract
(re-run ⇒ no dupes; removed source ⇒ edge retracted); Route53-zone linkage
into the DNS-dependency graph; hybrid impact/reachability query extension;
D5 statement: never Neo4j-only state, `neo4j-rebuild-bite.sh` compatibility
is a W3 exit criterion; eval thresholds (precision/recall, PROPOSED) that
W5-T2 will enforce.

**ADR-0059 — durable dispatch.** Outbox table shape + state machine; the
atomicity contract (outbox row committed in the same transaction as the
report-run state transition, both request paths); relay/recovery semantics
with every crash window enumerated and its resolution (at-least-once relay +
idempotent dispatch ⇒ no drop, no double-dispatch); relay metrics; the
platform `send_task` site inventory + wrapper mandate; ratchet mechanism
(static gate) + its planted-site negative control; rollout order.

**ADR-0060 — scale certification methodology.** Target table = the five §11
G-SCA bullets; harness composition (500-device seeded-estate generator,
100-user k6 profile, 5k-device/100k-interface projection fixture, 10× burst
profile, PgBouncer probes) parameterized by scale point; reduced-scale
execution posture + achieved-scale-point recording format; G-SCA re-base
mechanics on the Consultant scale-targets answer; the GA promotion trigger
(what happens if a certified-scale environment materializes); user
ratification of the §0 posture recorded in the ADR.
