## SLO

| Field | Value |
|---|---|
| §6 SLI | API availability (non-5xx on `/api/v1/*`) |
| §6 SLO | >= 99.9% / 30 days |
| Error budget | 1 - 0.999 = **0.1%** of requests |
| Recording rule (W3-T2) | `slo:netops_api_availability:ratio_rate5m` |
| Alerts (W3-T3) | `NetopsApiAvailabilityFastBurn` (page, 14.4x over 5m & 1h), `NetopsApiAvailabilitySlowBurn` (ticket, 6x over 30m & 6h) |
| ADRs | ADR-0046 §2 (multi-window burn-rate), ADR-0015 (alerting), ADR-0019 §4 (this runbook's narrative owner) |

## What firing means

The non-5xx request ratio is consuming the 0.1% / 30-day error budget faster than
the budget allows. The **fast (page)** tier means a real fast burn confirmed over
both a 5m and a 1h window; the **slow (ticket)** tier means a sustained low-grade
burn over 30m and 6h. Multi-window (fast + slow) is by design — a single 5m spike
does not page (ADR-0046 §2, alternatives 1/2).

## On-call checklist (deterministic)

1. Confirm the burn: `sum(rate(netops_http_requests_total{status_class="5xx"}[5m]))`
   vs total — which routes/status codes dominate the 5xx?
2. Check api pod health (HPA/PDB, W2-T1) and recent rollouts (a bad deploy is the
   most common cause).
3. Check the data tier the api depends on: Postgres (CloudNativePG, ADR-0042) and
   PgBouncer reachability, Redis Sentinel (ADR-0044) for session/cache.
4. If a deploy regression: roll back per the rolling-upgrade order (PRODUCTION.md
   §9). If a dependency outage: see the relevant DR runbook (`dr-postgres-pitr.md`,
   `dr-full-platform.md`).
5. After recovery, confirm the SLI recovers above objective and the alert clears.
